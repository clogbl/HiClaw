# Changelog (Unreleased)

Record image-affecting changes to `manager/`, `worker/`, `openclaw-base/` here before the next release.

---

- refactor(controller): replace the "inherit ownerReferences from the controller's own Pod" logic in `hiclaw-controller/internal/backend/kubernetes.go` with a standard CR-to-Pod controller OwnerReference stamped via `controllerutil.SetControllerReference`. `K8sBackend` now takes a `*runtime.Scheme`; `CreateRequest.Owner` carries the owning CR (Worker / Team / Manager), which the Worker / Team / Manager reconcilers populate. Finalizers still call `backend.Delete` to close the window between "CR deleted" and "container gone" — the ownerRef is purely additive safety net that lets K8s GC clean up the Pod even when a finalizer is patched off, and it deliberately does NOT replace the explicit delete because (a) the embedded deployment runs only `kube-apiserver` + `kine` (no kube-controller-manager → no GC controller, see `hiclaw-controller/internal/apiserver/embedded.go`) and (b) docker containers are never represented as Pod objects the apiserver could GC. `PodOverlay.OwnerReferences`, `ownerRefsCache`, `controllerOwnerRefs`, `getCurrentPod`, `filterOutReplicaSetOwners`, and `classifyAPIError` are removed.
- refactor(controller): rewrite `HumanReconciler` to align with the `Worker`/`Manager`/`Team` service-layer pattern. Introduces `service.HumanProvisioner` (implemented by `*service.Provisioner` in `provisioner_human.go`) so the reconciler no longer depends on `matrix.Client` directly; splits `human_controller.go` into phase files (`human_scope.go`, `human_reconcile_{infra,rooms,legacy,delete}.go`); replaces the imperative `handleCreate` / `handleUpdate` branches with a single declarative desired-vs-observed convergence loop; unifies status writes behind a defer-patch using `client.MergeFrom` + `Status().Patch`; skips `EnsureUser` on steady-state reconciles so `!admin users reset-password` no longer clobbers user-initiated password changes in Element (falls back to admin-only invite when login fails). **Matrix `LoginAsHuman` is now lazy**: `reconcileHumanInfra` no longer issues a Login on every 5-minute requeue (which would have created a fresh Matrix device session each time since `POST /_matrix/client/v3/login` without `device_id` never reuses sessions — ~288 devices/human/day); `ensureUserToken` only fires Login when `reconcileHumanRooms` actually needs a user token to `/join` a new room, reusing the token seeded by `EnsureHumanUser` on first-time provisioning and caching it across subsequent invite/join iterations in the same reconcile. Keeps the pre-refactor `"human created"` Info log in `reconcileHumanInfra` (in the first-time provisioning branch, gated on `Status.MatrixUserID == ""`) — `tests/test-19-human-and-team-admin.sh` greps `hiclaw-controller-error.log` for `"human created.*<name>"` to detect successful provisioning, and this also parallels `"worker created"` / `"manager created"` emitted by the other reconcilers. Drops the redundant `V(1).Info("human reconciled")` that only fired at debug level. Adds `test/testutil/mocks/MockHumanProvisioner` and `human_controller_test.go` with seven cases — create (asserts single-reconcile convergence + finalizer added + zero Login calls) / add-room / revoke-room / pending-resource / steady-state-no-Login (locks in the device-bloat fix) / delete / stale-password. Behavior preserved: no CRD changes, no new password-management features.
- feat(controller): align cloud-mode STS defaults for team members with the embedded MinIO policy and open up per-member customization. Team leaders and team workers now default to a single object-storage entry covering `agents/<name>/*` + `shared/*` + `teams/<team>/*` (all RW), mirroring `oss.MinIOAdminClient.buildWorkerPolicy` when `teamName != ""`; leader and team workers share the same scope (no cross-member elevation). `LeaderSpec` and `TeamWorkerSpec` gain an optional `accessEntries` field (same semantics as `WorkerSpec.AccessEntries`) so operators can override the defaults per-member; both CRD copies (`hiclaw-controller/config/crd/teams.hiclaw.io.yaml` and `helm/hiclaw/crds/teams.hiclaw.io.yaml`) have been updated. The `accessresolver` grows a `resolveTeamMember` code path routed from `ResolveForCaller` whenever `caller.Team != ""`, a `${self.team}` template variable, and a `DefaultEntriesForTeamMember()` default set. Standalone Worker defaults were also realigned: `shared/*` is now RW (previously read/list only) to match the embedded policy. Unit coverage in `internal/accessresolver` and `internal/credentials` updated accordingly.
- fix(controller): drop hard-coded `admin`/`admin` fallback credentials flagged by security scanners. `gateway.NewHigressClient` no longer rewrites empty `AdminUser` / `AdminPassword` to `"admin"`, and `config.LoadConfig()` no longer supplies `"admin"` as the default for `HICLAW_ADMIN_USER` / `HICLAW_ADMIN_PASSWORD` in `HigressAdminUser` / `HigressAdminPassword`, `MatrixAdminUser` / `MatrixAdminPassword`, or `WorkerEnv.AdminUser` — the install / deploy layer is now the sole source of these values, and a missing env var fails fast at login instead of silently running as `admin`/`admin`. The embedded-mode one-shot `admin`/`admin` bootstrap path inside `HigressClient.ensureSession` (guarded by `AllowDefaultAdminFallback`) is preserved, since it is the legitimate way to re-key a freshly-booted Higress instance.

- fix(controller): stop wiping AI-route `allowedConsumers` on controller restart. `gateway.HigressClient.EnsureAIRoute` now GETs `/v1/ai/routes/{name}` first and returns a no-op when the skeleton already exists (logging a WARN only if `pathPredicate` or upstream provider diverges) — it never writes `authConfig.allowedConsumers` in any branch, so the Initializer can no longer reset that field to `[]` on every boot and transiently lock Manager/Workers out with 403s. The `AllowedConsumers` field is removed from `gateway.AIRouteRequest` and from the Initializer's `EnsureAIRoute` call, cementing the ownership split: Initializer owns the route skeleton, `AuthorizeAIRoutes` / `DeauthorizeAIRoutes` own `allowedConsumers`. Three related hardening changes ship alongside: `modifyAIRoutes` now captures the first per-route GET/PUT error via a `firstErr`/`lastErr` pattern and returns it to the caller instead of silently `break`ing the retry loop and returning `nil`; `manager_reconcile_infra.reconcileManagerInfrastructure` surfaces `EnsureManagerGatewayAuth` errors as reconcile errors (previously logged as `(non-fatal)`) so controller-runtime re-queues with backoff; and the worker restart path in `ReconcileMemberInfra` now defensively calls a new `Provisioner.EnsureWorkerGatewayAuth` (mirroring the Manager method, using `consumer = "worker-<name>"`) after `RefreshCredentials`, so existing workers get re-bound to AI routes on controller startup instead of depending on the next spec-change event. New `gateway.client_test.go` cases (`TestAuthorizeAIRoutes_PUTFailsPropagatesError`, `TestEnsureAIRoute_ExistingSkipsPOST`, `TestEnsureAIRoute_MissingCreatesSkeletonWithoutAllowedConsumers`) lock in the contract. `MockProvisioner` gains the matching `EnsureWorkerGatewayAuthFn` hook.
- feat(controller,image): make the bundled `hiclaw` CLI usable directly inside the `hiclaw-controller` container — `docker exec hiclaw-controller hiclaw get managers default -o json` now works out of the box, matching the existing UX on the manager container. Three coupled changes:
  - `internal/service/provisioner_sa.go` adds `EnsureAdminServiceAccount` and `RequestAdminSAToken`, mirroring the existing `EnsureManagerServiceAccount` / `RequestManagerSAToken` pair (same SA name resolution via `prefix.AdminName()`, same long-lived `expSeconds=315360000` audience-bound `TokenRequest`). The admin SA name (`hiclaw-admin` under the default prefix) was already recognized by `prefix.ParseSAUsername` as `RoleAdmin`; this just wires up the missing create-and-mint primitives.
  - `internal/app/app.go::Start` calls a new `bootstrapAdminCLIToken` helper from inside the post-`Elected()` goroutine (right after `init.Run`, embedded mode only), which (a) ensures the admin SA exists, (b) mints a long-lived token via the embedded apiserver's TokenRequest API, and (c) writes it to `/var/run/hiclaw/cli-token` (mkdir 0700, file 0600). Embedded-only because incluster controllers usually lack the RBAC to create SAs / mint arbitrary SA tokens, and operators there have kubectl + their own creds anyway. Failures are logged loudly but non-fatal — the controller is still fully functional, only the in-container CLI sugar is degraded.
  - Both `hiclaw-controller/Dockerfile` and `Dockerfile.embedded` set `ENV HICLAW_CONTROLLER_URL=http://127.0.0.1:8090` and `HICLAW_AUTH_TOKEN_FILE=/var/run/hiclaw/cli-token` so the bundled CLI's `discoverToken()` (already env-driven) auto-loads the token without any per-call flags. Both vars are pure CLI defaults — neither the controller binary nor supervisord reads them, and `docker exec -e HICLAW_AUTH_TOKEN=…` still overrides cleanly. Path lives under `/var/run` (tmpfs on most container runtimes) so the token rotates naturally on every container start; this also means a brief window after container start where `docker exec … hiclaw` may 401 before bootstrap completes — acceptable, same race window already exists for the manager container.
  - Why not reuse `/data/admin-token`? That file holds the embedded **kube-apiserver**'s static-token-file token whose username surfaces as `admin` (group `system:masters`), which `prefix.ParseSAUsername` rejects with "unexpected username format" because it doesn't start with `system:serviceaccount:` — same root-cause as the earlier `HTTP 401: invalid or missing bearer token` that motivated the install script switching its welcome-wait poll from `hiclaw-controller` to `hiclaw-manager`. The new admin SA token IS a `system:serviceaccount:…` JWT and goes through the same TokenReview path that already works for managers and workers.

- fix(test): close the data race in `MockProvisioner.MatrixUserID` that was making `TestWorkerStateChange_StopAndResume` (and any other worker/team test that runs after a previous test's tear-down delete) flake under `-race`. `Reset()` writes `MatrixUserIDFn = nil` while holding `m.mu`, but `MatrixUserID()` was reading the same field without the lock — so the next test's `resetMocks()` would race with a still-in-flight `WorkerReconciler.reconcileDelete()` from the prior test (the standard "previous reconcile loop hasn't drained yet by the time the next test starts" pattern controller-runtime tests all hit). Visible in CI as `WARNING: DATA RACE` between `mocks/provisioner.go:69` (Reset write) and `mocks/provisioner.go:235` (MatrixUserID read), then `--- FAIL: TestWorkerStateChange_StopAndResume` even though the test logic itself was fine. Wrap the field read in the same `m.mu.Lock()` / `m.mu.Unlock()` envelope every other method in the file already uses; the lock is held only across the pointer copy, never across the user-supplied `Fn` callback, so no deadlock surface is added. Pre-existing on `main` (no PR-side mock changes touch this method); fixing here so the otherwise-unrelated welcome-wait PR isn't blocked by the flake.

- feat(install,controller,cli): block `install/hiclaw-install.{sh,ps1}` on the first-boot welcome message actually being delivered, so the installer no longer exits while the manager DM is still empty. Three coupled changes: (1) `internal/server/types.go::ManagerResponse` adds `WelcomeSent bool` (always serialized — `json:"welcomeSent"`, no `omitempty` — so the install script can rely on a stable JSON shape regardless of false-vs-unset), wired in `resource_handler.go::managerToResponse` from `m.Status.WelcomeSent`; this exposes the existing reconciler-owned idempotency flag on the existing `GET /api/v1/managers/{name}` endpoint with no new authz surface. (2) `cmd/hiclaw/get.go` mirrors the field on `managerResp` and prints it in the human-friendly `hiclaw get managers <name>` detail output, so admins can also inspect it manually. (3) Both installers, immediately after their existing `Wait-ManagerReady` / `_state == running` step (embedded mode only — the legacy single-container path already sent the welcome itself), poll `docker exec hiclaw-controller hiclaw get managers default -o json` every 3s and look for `"welcomeSent":true` (whitespace-stripped via `tr -d` / `-replace '\s'` so the grep is independent of go-json field ordering or future formatting). The poll runs inside `hiclaw-controller`, the source-of-truth container: its bundled `hiclaw` CLI binary is always in lockstep with the controller binary serving the HTTP API (same `go build` output), so when the controller adds a new field like `welcomeSent` the CLI that parses the response is guaranteed to know about it on the same image. The `hiclaw-manager` container ships its own (potentially-older) CLI binary frozen at the manager image's build time — running the poll there silently dropped the new field across upgrades and left the install loop hung. The controller container mints a long-lived admin SA token at startup and writes it to `HICLAW_AUTH_TOKEN_FILE=/var/run/hiclaw/cli-token` (set as a Dockerfile ENV default), so the bare `docker exec hiclaw-controller hiclaw …` form auto-discovers both endpoint and token. (Earlier iterations tried `hiclaw-manager` with its `HICLAW_AUTH_TOKEN` SA-JWT, and before that `docker exec -e HICLAW_AUTH_TOKEN_FILE=/data/admin-token hiclaw-controller`; the latter failed with `HTTP 401: invalid or missing bearer token` because `/data/admin-token` is the embedded **kube-apiserver**'s static-token-file token whose username surfaces as `admin` — `prefix.ParseSAUsername` rejects anything not starting with `system:serviceaccount:`. The new in-container admin SA fixes this end-to-end.) Default timeout 300s (overridable via `HICLAW_WELCOME_TIMEOUT`); the timeout path is deliberately non-fatal — install is still functional even when the welcome hasn't shipped (services are all up, the silence is just in the DM), so on timeout the installer prints a three-line warning block that (a) restates that the install succeeded, (b) tells the admin how to recover by hand — log into Element Web (the URL/credentials are in the success banner immediately below the warning), open the DM with the Manager, and send any message to make the agent take over and start the guided setup, and (c) points at `docker exec hiclaw-controller hiclaw get managers default` for inspection. New message keys `install.welcome_msg.timeout_hint` / `install.welcome_msg.timeout_inspect` (zh+en) keep the recovery instructions localized. When the in-container `hiclaw` CLI isn't found (very old controller image without this CLI built in — should not happen in practice since the CLI has always been bundled), we log a one-line note and skip the wait so legacy images still install.

- fix(install): `hiclaw-install.sh uninstall` (and the PowerShell `Uninstall-HiClaw` equivalent) now also stop and remove the `hiclaw-controller` container, so the `hiclaw-data` Docker volume can actually be reclaimed. Symptom: after running `bash install/hiclaw-install.sh uninstall` and re-installing, the Admin DM Room and all its message history were still present — the new install was just attaching to the same Tuwunel database. Root cause: when the embedded architecture landed in PR #616 the long-running container was renamed `hiclaw-docker-proxy` → `hiclaw-controller` and gained a `${HICLAW_DATA_DIR}:/data` mount that holds Tuwunel's state (rooms, users, tokens), MinIO state, and Higress state, but `uninstall_hiclaw()` / `Uninstall-HiClaw` were never updated to know about the new container name. The volume removal step (`docker volume rm hiclaw-data`) then silently failed because the controller container was still attached, swallowed by the existing trailing `|| true` / `*>$null`. Both installers now stop and remove `hiclaw-controller` between the legacy-proxy cleanup block and the volume removal block, with a new `uninstall.stopping_controller` (zh+en) log line that calls out the container's role ("内嵌 Tuwunel/MinIO/Higress") so the cleanup is visible. The reinstall path was already correct (it always cleaned `hiclaw-controller` — see `step_existing` in the bash script and the equivalent PowerShell branch); only the standalone `uninstall` subcommand was missing the cleanup. The Makefile target `make uninstall-embedded` was also already correct as a workaround. No changes to message keys other than the additive `uninstall.stopping_controller`, so no other call sites need updating.

- fix(controller): gate the first-boot welcome prompt on `Provisioner.IsManagerLLMAuthReady` in addition to the existing `IsManagerJoinedDM` check, so the prompt only ships once the manager's bearer token can actually drive an end-to-end LLM call through the gateway. Symptom that motivated the fix: on a fresh install the manager Matrix user auto-joins the Admin DM in ~5-10s and the controller would immediately deliver the welcome — but Higress's WASM key-auth filter takes ~40-45s to sync the freshly-bound consumer onto the AI route (the gap the legacy `start-manager-agent.sh` papered over with `sleep 45` after `setup-higress.sh`), so the manager's first `/v1/chat/completions` reply attempt 401s against the gateway and the entire onboarding turn is silently lost. Three coupled changes restore correctness without re-introducing the legacy sleep: (1) `service.ProvisionerConfig`/`Provisioner` gain `AIGatewayURL` AND `ManagerModel` (wired in `internal/app/app.go` from `cfg.WorkerEnv.AIGatewayURL` and `cfg.ManagerModel` respectively — both already in `Config`, so no new install-time surface), and a new `Provisioner.IsManagerLLMAuthReady(ctx, gatewayKey)` POSTs a real chat-completions request `{"model":"<ManagerModel>","messages":[{"role":"user","content":"Reply with only one word: ok"}]}` to `<AIGatewayURL>/v1/chat/completions` with the manager's bearer token. This is the same code path the manager itself will exercise on its first reply, so a successful probe is end-to-end proof-of-life — auth filter, route, upstream provider, and model resolution — rather than a synthetic auth-only check. Probe body deliberately uses only the universally-supported `model` and `messages` fields (no `max_tokens`: some openai-compat providers, notably Bedrock-fronted models and o1/o3-style reasoning families, reject the parameter outright with 400 — readiness would never go true on those backends). The brevity-instructed prompt typically yields a 1-2 token reply so per-probe cost is ~10-20 tokens total round-trip, negligible across the handful of probes issued during the propagation window. Status-code policy: HTTP 200 → ready; 401/403 → "not ready, requeue silently" (expected during the propagation window — caller emits a V(1) line, no WARN spam); anything else (400, 404, 429, 5xx) or network/dial error → returned as `error` so the reconciler logs at WARN and the operator can spot persistent misconfigurations (wrong model name, upstream down, quota exhausted). The welcome simply stays unsent until the call returns 200, which matches user intent: better a delayed welcome than one the manager cannot answer. Empty `AIGatewayURL` / `ManagerModel` / `gatewayKey` → return `(true, nil)` so unit tests and bring-your-own-gateway deploys do not stall the welcome forever. (2) `service.ManagerProvisioner` interface + `testutil/mocks/manager_provisioner.go` add `IsManagerLLMAuthReady` (with `IsManagerLLMAuthReadyFn` hook + `LLMAuthReadyCallCount` accessor); `IsManagerJoinedDM` is preserved unchanged so the welcome reconcile gates on BOTH signals. (3) `internal/controller/manager_reconcile_welcome.go` polls both checks BEFORE the `Status().Patch` claim — either one returning false short-requeues without touching status, so we don't churn `WelcomeSent` with claim/rollback patches while waiting for auth to propagate. New integration test `TestManagerWelcome_LLMAuthNotReady_RequeuesUntilPropagated` mirrors the existing membership-poll test: drive `IsManagerLLMAuthReady` to return false twice then true, assert the controller never calls `SendManagerWelcomeMessage` until the auth signal flips, assert exactly one send after it does, and that the manager phase stays `Running` throughout the requeue loop.

- feat(controller,manager): restore the first-boot welcome / onboarding prompt that the legacy `start-manager-agent.sh` used to send in single-container deploys, but for the new embedded / k8s mode where the prompt was silently lost. The legacy script's heredoc-driven `welcome` block is gated behind `[ "${HICLAW_RUNTIME}" != "k8s" ]` (manager containers in embedded mode intentionally don't have the admin password — only `HICLAW_ADMIN_USER` — so they could not log in as admin to send the prompt themselves), and `service.Provisioner.ProvisionManager` Step 4 only created the Admin DM Room without ever delivering the message. As a result, every fresh embedded install booted into a silent admin DM and the Manager Agent waited for the admin to type the first message instead of running its identity-setup four-question flow. Five coupled changes restore the behavior end-to-end and make the new owner the controller (which already holds admin Matrix credentials in `TuwunelClient.ensureAdminToken` and the room ID in `Manager.Status.RoomID`):
  - `api/v1beta1/types.go` + `config/crd/managers.hiclaw.io.yaml` + `helm/hiclaw/crds/managers.hiclaw.io.yaml`: add `ManagerStatus.WelcomeSent bool`, the idempotency guard. Set once on first successful send and never cleared, so manager container restarts / spec changes / leader-election handoffs never re-prompt the admin DM.
  - `internal/matrix/client.go`: extend the `matrix.Client` interface with `SendMessageAsAdmin(ctx, roomID, body)`; the `TuwunelClient` impl reuses the cached admin token from `ensureAdminToken` so the welcome path doesn't need a new auth flow.
  - `internal/service/provisioner.go`: add `ManagerWelcomeRequest{RoomID, Language, Timezone}` and `Provisioner.SendManagerWelcome(ctx, req) (sent bool, err error)`. The method first verifies the manager Matrix user has actually joined the DM room (returns `(false, nil)` if not, signaling "requeue, don't fail"), then renders the prompt via `renderManagerWelcomeBody(language, timezone)` — which mirrors the legacy heredoc text — and sends as admin. The membership pre-check exists because OpenClaw / hermes-agent's first `/sync` auto-join is a few seconds after the container reports Running; sending before then makes the message land as a historical event the agent may not act on.
  - `internal/controller/manager_reconcile_welcome.go` (new): inserts a `reconcileManagerWelcome` sub-step after `reconcileManagerContainer` and before status logging. Skip cases (early-return, no requeue): `WelcomeSent` already true, `RoomID` not yet provisioned, or container Status is not Running/Ready. While waiting for the agent's first `/sync` to auto-join the DM room it polls `Provisioner.IsManagerJoinedDM` (side-effect-free, split out from `SendManagerWelcome` precisely so this poll does NOT churn `Status.WelcomeSent` with claim/rollback patches on every requeue) and returns `RequeueAfter: 5s` until membership lands. Once joined it CLAIMS the slot via an inline `r.Status().Patch` against a deep-copy of the manager (the deep copy is critical — patching the working `m` directly would let the API-server response wipe in-memory status fields like `RoomID` that earlier reconcile steps set but that have not yet been persisted by the deferred end-of-Reconcile patch); a Conflict on the claim means another reconcile won the race and we exit silently without sending. Only after the claim is committed does it call `Provisioner.SendManagerWelcomeMessage`. This claim-before-send ordering is what makes "exactly one welcome per manager lifecycle" deterministic even when two reconciles read `WelcomeSent=false` from a stale informer cache (which actually happens in CI: the post-Create reconcile starts ~3ms after the previous reconcile's deferred status patch, before the cache has observed it). On send-after-claim error we log loudly and return nil — we deliberately do not roll back the claim because silent re-attempt would re-open the race, and a missing welcome can be recovered by the admin prompting manually whereas a permanent double welcome cannot. On error it logs and requeues but never flips the manager phase to Failed — a silent welcome is recoverable, a wedged manager is not.
  - `internal/config/config.go` + `internal/app/app.go`: read `HICLAW_LANGUAGE` / `TZ` env vars (defaults `zh` / `Asia/Shanghai` to match the legacy heredoc's defaults so observable behavior is identical when an admin upgrades from single-container) and wire them through `Config.UserLanguage` / `Config.UserTimezone` to `ManagerReconciler.UserLanguage` / `ManagerReconciler.UserTimezone`, where `reconcileManagerWelcome` forwards them into `ManagerWelcomeRequest`. Also `manager/scripts/init/start-manager-agent.sh`: expand the previous one-liner k8s-mode skip comment to spell out that the controller now owns both DM-room creation AND the welcome message, and explain why the manager container could not own this even if we wanted it to (no admin password). Tests: `internal/matrix/client_test.go` adds `TestSendMessageAsAdmin` covering admin-token reuse + payload routing; `test/integration/controller/manager_test.go` adds `TestManagerWelcome_HappyPath_SendsOnce` (single send + correct language/timezone forwarding, with the suite wiring `en` / `America/Los_Angeles` to assert non-default values reach the provisioner — this is the test that originally caught the cache-lag race that motivated the claim-before-send pattern), `TestManagerWelcome_Idempotent_NoResendOnReconcile` (no resend after `WelcomeSent=true` even when a fresh reconcile is forced), and `TestManagerWelcome_NotJoinedYet_RequeuesUntilJoined` (`IsManagerJoinedDM` returns false twice then true; controller must keep the manager in `Running` phase across the membership-poll loop, must NOT call `SendManagerWelcomeMessage` until membership lands, and must only flip `WelcomeSent` after the successful call). Mock surface gains `IsManagerJoinedDMFn` + `SendManagerWelcomeMessageFn` plus thread-safe `WelcomeCallCount` / `WelcomeCallsSnapshot` / `ClearWelcomeCalls` / `JoinedDMCallCount` accessors so tests can read recorded calls without touching the mock's lock directly.

- fix(controller): `hiclaw create worker` / `hiclaw apply worker` no longer hard-code `qwen3.5-plus` as the fallback model when `--model` is omitted. Both `cmd/hiclaw/create.go::createWorkerCmd` and `cmd/hiclaw/apply.go::{applyWorkerZip,applyWorkerParams}` now route through a new shared `defaultWorkerModel()` helper that prefers `$HICLAW_DEFAULT_MODEL` (the install-time choice that the controller already injects into every Manager container via `service.WorkerEnvBuilder.BuildManager`) and only falls back to `qwen3.5-plus` when the env var is unset or whitespace. Without this, a Manager Agent that ran `hiclaw create worker --name alice` (no `--model` flag — the documented and skill-default invocation in `manager/agent/skills/worker-management/references/create-worker.md`) silently overrode the admin's install-time model selection (e.g. `qwen3-coder`) with `qwen3.5-plus`, leaving every spawned Worker on the wrong LLM. CLI `--help` text and the related skill docs (`worker-management/references/create-worker.md`, `team-management/references/create-team.md`) now describe the env-var-aware default; new `TestDefaultWorkerModel` table covers the env-set / env-empty / whitespace-only cases.

- fix(install,build): publish the `hiclaw-embedded` all-in-one image and stop the installer from silently falling back to the broken legacy single-container path. Since PR #616 the lightweight `hiclaw-manager` image no longer bundles Higress / Tuwunel / MinIO / Element Web — those services live exclusively in `hiclaw-embedded`, which is built from `hiclaw-controller/Dockerfile.embedded`. But `Makefile` only had a local `build-embedded` target (used by `make install-embedded` and integration tests), and neither `.github/workflows/build.yml` nor `.github/workflows/build-rc.yml` ever pushed the embedded image to the registry. As a result, every public install of v1.1.0-rc.1 (and any prior tag whose manager was already slim) hit `install/hiclaw-install.sh::resolve_embedded_image`, failed to pull `higress/hiclaw-embedded:${VERSION}` *and* `:latest` ("manifest unknown"), silently set `HICLAW_USE_EMBEDDED=0`, and spun up the legacy single-container `hiclaw-manager`, which then looped forever on `[hiclaw] Waiting for Higress Gateway at 127.0.0.1:8080... ERROR: did not become available within 180s` because the image no longer ships those services. Four coupled changes fix the chain end-to-end:
  - `Makefile`: new `EMBEDDED_IMAGE` / `EMBEDDED_TAG` variables and a `push-embedded: push-hiclaw-controller buildx-setup` target that builds `hiclaw-controller/Dockerfile.embedded` for `linux/amd64,linux/arm64` and pushes a multi-arch manifest to `${REGISTRY}/${REPO}/hiclaw-embedded:${VERSION}`. The dependency on `push-hiclaw-controller` ensures the inner `COPY --from=${HICLAW_CONTROLLER_IMAGE}` stage resolves against a freshly pushed registry tag (passed in via `--build-arg HICLAW_CONTROLLER_IMAGE=$(CONTROLLER_TAG)`).
